## Goal

## Constraints

- **Prisma 7** (the spec is explicit: read the v7 upgrade guide; many v6 patterns break)
- **Neon PostgreSQL** (already provisioned by user; dev + prod branches; user supplies `DATABASE_URL`)
- **Never `prisma db push`** â€” always `prisma migrate dev` per `context/coding-standards.md:77`

## Prisma 7 Breaking Changes That Drive This Design

These are the v7 changes that aren't optional. If we skip any, the implementation breaks.

| Change | Impact on us |
|---|---|
| ESM-only package | `package.json` must have `"type": "module"` |
| New `prisma-client` generator (replaces `prisma-client-js`) | Schema generator block must use `provider = "prisma-client"` |
| `output` field is **required** | Client is no longer auto-emitted to `node_modules/@prisma/client`. We choose `src/generated/prisma` |
| Imports change | `import { PrismaClient } from "@/generated/prisma/client"` instead of `"@prisma/client"` |
| Driver adapter is **required** for all databases | Need `@prisma/adapter-neon` + `@neondatabase/serverless` |
| `datasource.url` is deprecated in schema | Move to `prisma.config.ts` via `env("DATABASE_URL")` |
| Env vars not auto-loaded | Add `dotenv` and `import "dotenv/config"` in `prisma.config.ts` and `lib/prisma.ts` |
| `migrate dev` no longer auto-runs `generate` or `seed` | Add explicit `db:*` npm scripts |
| `--skip-generate`, `--skip-seed`, `--from-url`, `--to-url`, `--schema` flags removed | Don't use them in scripts/docs |
| Client middleware (`prisma.$use`) removed | N/A (not using) â€” use Client Extensions if needed later |
| Min Node 20.19, TS 5.4 | Already satisfied |
| SSL cert validation stricter | Neon uses valid certs, so default is fine â€” no `rejectUnauthorized: false` needed |

## Tech Stack

| Layer | Choice |
|---|---|
| ORM | Prisma 7 (`prisma@7`, `@prisma/client@7`) |
| Driver adapter | `@prisma/adapter-neon` |
| Underlying driver | `@neondatabase/serverless` |
| Env loader | `dotenv` |
| Seed runner | `tsx` (per Prisma 7 quickstart) |
| Database | Neon PostgreSQL (dev branch URL in `.env`, prod URL injected at deploy) |

**Why `@prisma/adapter-neon` (decision)**: Neon's native WebSocket/HTTP driver. Works in both Node and Edge runtimes, so future NextAuth middleware or Edge routes can query the DB. Trade-off vs `@prisma/adapter-pg`: tiny extra setup, but no runtime lock-in.

## Schema (initial)

Source of truth: `context/project-overview.md:197-389`. We copy the schema verbatim with **two changes only**:

1. Replace the generator block with the Prisma 7 version
2. Remove `url` from the datasource block (moved to `prisma.config.ts`)

```prisma
generator client {
  provider = "prisma-client"
  output   = "../src/generated/prisma"
}

datasource db {
  provider = "postgresql"
  // URL configured in prisma.config.ts
}

// Models below are unchanged from project-overview.md:
//   User, Account, Session, VerificationToken,
//   Item, ItemType, Collection, ItemCollection, Tag,
//   plus the ContentType enum.
// Indexes and onDelete: Cascade rules are preserved.
```

Models include (from `project-overview.md`):

- **User** â€” `cuid()` id, email unique, optional `password`, `isPro`, Stripe fields, NextAuth relations
- **Account / Session / VerificationToken** â€” NextAuth v5 (database session) models, cascade delete on user
- **Item** â€” `cuid()` id, `ContentType` enum, optional content/file/url fields, isFavorite/isPinned, language; indexes on `userId`, `itemTypeId`, `createdAt`
- **ItemType** â€” system or user-owned; unique `(name, userId)`
- **Collection** â€” user-owned, optional `defaultType`; index on `userId`
- **ItemCollection** â€” join table, composite `@@id`, cascade both sides
- **Tag** â€” many-to-many via implicit `"ItemTags"` relation

All `onDelete: Cascade` and `@@index` from the original schema are preserved.

## Singleton Prisma Client

```ts
// src/lib/prisma.ts
import "dotenv/config";
import { PrismaNeon } from "@prisma/adapter-neon";
import { PrismaClient } from "@/generated/prisma/client";

const globalForPrisma = globalThis as unknown as {
  prisma?: PrismaClient;
};

function createPrismaClient() {
  const connectionString = process.env.DATABASE_URL;
  if (!connectionString) {
    throw new Error("DATABASE_URL is not set");
  }
  return new PrismaClient({
    adapter: new PrismaNeon({ connectionString }),
  });
}

export const prisma = globalForPrisma.prisma ?? createPrismaClient();

if (process.env.NODE_ENV !== "production") {
  globalForPrisma.prisma = prisma;
}
```

The dev-mode HMR guard is necessary in Next.js so we don't spawn a new client on every hot reload.

## prisma.config.ts

```ts
import "dotenv/config";
import { defineConfig, env } from "prisma/config";

export default defineConfig({
  schema: "prisma/schema.prisma",
  migrations: {
    path: "prisma/migrations",
    seed: "tsx prisma/seed.ts",
  },
  datasource: {
    // DIRECT_URL = unpooled Neon endpoint. The Prisma CLI (migrate/db push/
    // introspect) needs a direct TCP connection. The pooled DATABASE_URL is
    // used at runtime by @prisma/adapter-neon over WebSocket.
    url: env("DIRECT_URL"),
  },
});
```

**Neon two-URL pattern (important).** Neon requires *two* connection strings:

- `DATABASE_URL` â€” pooled (`-pooler` in hostname). Used at runtime by `@prisma/adapter-neon` over WebSocket. Optimal for serverless / many short-lived connections.
- `DIRECT_URL` â€” direct/unpooled. Used by the Prisma CLI (the Rust-based schema engine that runs migrations). The pooler endpoint can't serve direct TCP migration traffic.

Add `connect_timeout=15` to both so Neon's cold start (compute scale-to-zero) doesn't immediately fail.

## Seed Script

Seeds the 7 system `ItemType` rows from `project-overview.md:401-409`. Idempotent (uses `upsert` on the `(name, userId)` unique key with `userId: null`).

```ts
// prisma/seed.ts
import "dotenv/config";
import { PrismaNeon } from "@prisma/adapter-neon";
import { PrismaClient } from "../src/generated/prisma/client";

const adapter = new PrismaNeon({ connectionString: process.env.DATABASE_URL! });
const prisma = new PrismaClient({ adapter });

const systemItemTypes = [
  { name: "snippet", icon: "Code",       color: "#3b82f6", isSystem: true },
  { name: "prompt",  icon: "Sparkles",   color: "#8b5cf6", isSystem: true },
  { name: "command", icon: "Terminal",   color: "#f97316", isSystem: true },
  { name: "note",    icon: "StickyNote", color: "#fde047", isSystem: true },
  { name: "file",    icon: "File",       color: "#6b7280", isSystem: true },
  { name: "image",   icon: "Image",      color: "#ec4899", isSystem: true },
  { name: "link",    icon: "Link",       color: "#10b981", isSystem: true },
];

async function main() {
  for (const t of systemItemTypes) {
    await prisma.itemType.upsert({
      where: { name_userId: { name: t.name, userId: null } },
      update: {},
      create: t,
    });
  }
}

main()
  .catch((e) => { console.error(e); process.exit(1); })
  .finally(() => prisma.$disconnect());
```

## Workflow / Verification

1. User confirms `DATABASE_URL` is in `.env` (points to their Neon **dev** branch)
2. `npm install` (installs new deps)
3. `npm run db:migrate -- --name init` â†’ creates `prisma/migrations/<ts>_init/` on disk and applies to Neon dev branch
4. `npm run db:generate` â†’ produces `src/generated/prisma`
5. `npm run db:seed` â†’ upserts 7 system item types
6. `npm run db:status` â†’ confirms "Database schema is up to date"
7. Quick smoke test: a tiny `tsx scripts/smoke.ts` that runs `prisma.itemType.count()` and prints 7
8. `npm run build` must complete with zero errors
9. `npm run lint` must complete with zero errors

## Risks & Mitigations

| Risk | Mitigation |
|---|---|
| `"type": "module"` breaks existing config files | Project only has `.ts` and `.mjs` config files â€” both are unaffected. Verify with `npm run build` after change. |
| Generated client in `src/generated` accidentally committed | Add to `.gitignore` in the same commit as the schema |
| User runs `prisma db push` by accident | Already documented in `coding-standards.md`; we don't add any script that uses `db push` |
| Driver adapter connection pool defaults differ from v6 | Neon's pooled URL handles pooling at the proxy; PrismaNeon defaults match Neon's expectations |
| Seed script duplicates rows on rerun | `upsert` on `(name, userId)` unique key â€” idempotent |

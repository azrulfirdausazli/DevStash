# Auth Phase 1 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Wire NextAuth v5 with the Prisma adapter and GitHub OAuth to protect `/dashboard/*` routes, replacing the hardcoded `demo@devstash.io` lookup in `getCurrentUserId()` with a live session read.

**Architecture:** NextAuth v5 split-config pattern — `auth.config.ts` (edge-safe, providers only) is imported by both the Next.js 16 proxy (`src/proxy.ts`) and the full `src/auth.ts` (which adds the Prisma adapter + JWT strategy + id-propagation callbacks). The proxy matcher is restricted to `/dashboard/:path*` so `/api/auth/*` is never intercepted. The `getCurrentUserId()` API is preserved; only the body changes (wraps `auth()` instead of `prisma.user.findUnique`).

**Tech Stack:** Next.js 16.2, React 19, NextAuth v5 (`next-auth@beta`), `@auth/prisma-adapter`, Prisma 7 + Neon serverless adapter (existing).

**Source spec:** `context/features/auth-phase-1-spec.md`
**Design doc:** `docs/superpowers/specs/2026-06-17-auth-phase-1-design.md` (currently staged, uncommitted — review and commit alongside the plan if you choose)

- [ ] **Step 1: Create the type augmentation file**

Create `src/types/next-auth.d.ts`:

```ts
declare module "next-auth" {
  interface Session {
    user: {
      id: string;
    } & DefaultSession["user"];
  }
}

declare module "next-auth/jwt" {
  interface JWT {
    userId?: string;
  }
}
```

The file does not need to be imported anywhere — TypeScript picks it up automatically because it lives in the project and ends in `.d.ts`.

```bash

```

Expected: zero errors. (No `session.user.id` usage yet, but the augmentation itself must be valid.)

- [ ] **Step 3: Create the edge-safe auth config**

Create `src/auth.config.ts`:

```ts
import GitHub from "next-auth/providers/github";
import type { NextAuthConfig } from "next-auth";

export default {
  providers: [GitHub],
} satisfies NextAuthConfig;
```

No `NextAuth()` call. No adapter. No `pages` key (uses NextAuth's default signin page).

- [ ] **Step 4: Create the full auth config**

Create `src/auth.ts`:

```ts
import NextAuth from "next-auth";
import { PrismaAdapter } from "@auth/prisma-adapter";
import { prisma } from "@/lib/prisma";
import authConfig from "@/auth.config";

export const { handlers, auth, signIn, signOut } = NextAuth({
  adapter: PrismaAdapter(prisma),
  session: { strategy: "jwt" },
  ...authConfig,
  callbacks: {
    async jwt({ token, user }) {
      if (user) token.userId = user.id;
      return token;
    },
    async session({ session, token }) {
      if (token.userId) {
        session.user.id = token.userId as string;
      }
      return session;
    },
  },
});
```

The `jwt` callback fires when the JWT is created (right after signin); the `session` callback fires on every session read. This propagation is what makes `session.user.id` available everywhere.

- [ ] **Step 5: Create the API route handler**

Create `src/app/api/auth/[...nextauth]/route.ts`:

```ts
import { handlers } from "@/auth";

export const { GET, POST } = handlers;
```

Next.js requires the directory structure `src/app/api/auth/[...nextauth]/` for the catch-all to bind correctly. The `[...nextauth]` segment is literal (square brackets and three dots) — it's a Next.js rest-segment convention.

- [ ] **Step 6: Create the proxy**

Create `src/proxy.ts`:

```ts
import { auth } from "@/auth";

export const proxy = auth((req) => {
  const isDashboard = req.nextUrl.pathname.startsWith("/dashboard");
  if (isDashboard && !req.auth) {
    const signInUrl = new URL("/api/auth/signin", req.url);
    signInUrl.searchParams.set("callbackUrl", req.nextUrl.pathname);
    return Response.redirect(signInUrl);
  }
});

export const config = {
  matcher: ["/dashboard/:path*"],
};
```

The matcher restricts the proxy to `/dashboard/*`. NextAuth's `/api/auth/*` routes are never intercepted, so signin/signout/callback work without recursion.

- [ ] **Step 7: Verify everything compiles and the build still passes**

```bash

npx tsc --noEmit
npm run build
```

Expected:
- `npm run lint` — zero errors
- `npx tsc --noEmit` — zero errors (this confirms the module augmentation is wired correctly — `auth.ts` references `session.user.id` and `token.userId` which only type-check because of `src/types/next-auth.d.ts`)
- `npm run build` — completes successfully. The build step is what confirms Next.js detects the new route handler at `app/api/auth/[...nextauth]/route.ts` and the proxy at `src/proxy.ts`. If either is missing or misnamed, the build will warn.

## Task 3: Refactor `getCurrentUserId()` to wrap `auth()`

**Files:**
- Modify: `src/lib/db/user.ts` (entire file body, but keep the export name and signature)

- [ ] **Step 1: Replace the file contents**

Replace `src/lib/db/user.ts` with:

```ts
import { cache } from "react";
import { auth } from "@/auth";

export const getCurrentUserId = cache(async (): Promise<string> => {
  const session = await auth();
  if (!session?.user?.id) {
    throw new Error(
      "Unauthenticated request reached a server component. " +
        "The /dashboard/* proxy should have redirected to /api/auth/signin before this.",
    );
  }
  return session.user.id;
});
```

What changed vs. the previous body:
- Dropped: `import { prisma } from "@/lib/prisma"`, `DEMO_USER_EMAIL` constant, `prisma.user.findUnique` call, the "Run \`npm run db:seed\` to create it" error message.
- Added: `import { auth } from "@/auth"`, `await auth()`, the new error message.

What stayed the same:
- The `cache()` wrapper (preserves QW-2's dedup behavior — both `layout.tsx` and `page.tsx` share one `auth()` call per render).
- The export name `getCurrentUserId` and return type `Promise<string>` (no call-site changes in `src/app/dashboard/layout.tsx:12` or `src/app/dashboard/page.tsx:17`).
